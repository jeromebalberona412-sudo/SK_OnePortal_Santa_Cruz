**Plan**
1. Define the registration state machine first: barangay selected, form submitted pending verification, email verified, password set, and active account. This becomes the source of truth for UI transitions, controller behavior, and login eligibility.
2. Add the database structure before controller work. Reuse the existing tenant and barangay concepts from [SK_Oneportal.sql](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/database_structure/SK_Oneportal.sql#L1), then introduce a dedicated Kabataan onboarding table for the pending application. Recommended fields are `tenant_id`, `barangay_id`, applicant name fields, email, contact number, form payload, `status`, `submitted_at`, `email_verified_at`, `password_set_at`, `reviewed_by_user_id`, and a nullable `user_id` link for the final account. Add a separate verification-token table only if signed links are not enough for the flow.
3. Make barangay selection the authoritative entry point. The selected barangay from [signup.blade.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/KKProfiling/Views/signup.blade.php#L1) should be carried into the form route, validated against the tenant’s barangay list, and stored with the pending registration so the rest of the flow is tenant- and barangay-scoped.
4. Refactor KK Profiling submission so it persists data instead of flashing a temporary success message. The submit action in [KKProfilingController.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/KKProfiling/Controllers/KKProfilingController.php#L80) should create or update the pending registration, trigger email verification, and keep the existing form UX in [kkprofiling.blade.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/KKProfiling/Views/kkprofiling.blade.php#L1).
5. Replace the prototype email verification screens with a real verification step tied to the pending registration. The routing surface in [auth.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/Authentication/Routes/auth.php#L1) and the current verification views in [email-verification.blade.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/Authentication/Views/email-verification.blade.php#L1) and [verify-success.blade.php](/home/frankien11/Documents/GitHub/SK_OnePortal_Santa_Cruz/Kabataan/app/Modules/Authentication/Views/verify-success.blade.php#L1) should advance the same persisted registration record. Use a signed verification link or a hashed token so verification can work across devices.
6. Add password setup after verification and only then create or activate the final user account. The account should become login-eligible only when the password exists and the registration is marked complete.
7. Build the official review path so only the SK officials of the selected barangay can see the submitted form by default. Access must be scoped by `barangay_id` and role checks, so if the user selects Calios then only Calios SK officials can view that submission. Admin or SK-Federations access should be allowed only through explicit authorization rules.
8. Finish with tests and validation for the full path: barangay selection, form persistence, verification, password creation, account activation, and barangay-only visibility for submitted forms.
