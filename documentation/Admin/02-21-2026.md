## Overview

The Admin Web App implements a secure, multi-layered authentication system using Laravel Fortify. The login feature includes advanced security mechanisms such as brute-force protection, account lockout, mandatory two-factor authentication, comprehensive audit logging, and security headers.

### Key Features
- ✅ Email/Password authentication via Laravel Fortify
- ✅ Brute-force protection with rate limiting
- ✅ Account lockout mechanism (5 attempts, 15-minute lockout)
- ✅ IP-based and email-based attempt tracking
- ✅ Mandatory Two-Factor Authentication (2FA) with TOTP
- ✅ QR code generation for 2FA setup
- ✅ Recovery codes (10 codes per user)
- ✅ Comprehensive audit logging (security, audit, email events)
- ✅ Security headers (HSTS, CSP, X-Frame-Options, etc.)
- ✅ HTTPS enforcement in production
- ✅ Admin-only access control
- ✅ Session regeneration on login
- ✅ Generic error messages (no user enumeration)

## Security Features Implemented

### 1. Rate Limiting & Brute-Force Protection

**Implementation:**
- **File:** `app/Services/LoginSecurityService.php`
- **Mechanism:** Tracks failed login attempts by email and IP address
- **Thresholds:**
  - Maximum attempts: 5 failures within 15 minutes
  - Lockout duration: 15 minutes
  - Password reset threshold: 3 lockout cycles

**Key Methods:**
```php
checkAccountLockout(string $email, string $ip): ?Carbon
recordFailedAttempt(string $email, string $ip, string $userAgent): void
evaluateLockout(?User $user, string $email, string $ip): bool
```

### 2. User Enumeration Prevention

**Strategy:** Generic error messages for all authentication failures
- Invalid credentials → Generic error
- Account locked → Generic error
- Non-admin user → Generic error
- User doesn't exist → Generic error

**Implementation Location:** `app/Providers/FortifyServiceProvider.php`

```php
// All failures return null with same error message
if (!$user || !Hash::check($password, $user->password)) {
    // Record and return generic error
    return null;
}
```

### 3. Admin-Only Access Control

**Implementation:**
```php
$isAdmin = $user->getRawOriginal('is_admin');
if ($isAdmin !== null && ! (bool) $isAdmin) {
    // Log and deny access with generic error
    return null;
}
```

### 4. Session Security

**Features:**
- Session regeneration on successful login
- CSRF token regeneration on logout
- Session invalidation on logout

**Implementation:** `app/Http/Controllers/AuthController.php`

### 5. Security Headers

**File:** `app/Http/Middleware/SecurityHeaders.php`

**Headers Applied:**
```
Strict-Transport-Security: max-age=31536000; includeSubDomains
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; ...
```

### 6. HTTPS Enforcement

**File:** `app/Http/Middleware/ForceHttps.php`
- Automatic HTTPS redirect in production environment
- 301 permanent redirect status

### 7. Comprehensive Logging

**Three separate log channels:**

1. **Security Log** (`storage/logs/security.log`)
   - Failed login attempts
   - Account lockouts
   - Retention: 90 days

2. **Audit Log** (`storage/logs/audit.log`)
   - All authentication events
   - User actions
   - Retention: 365 days

3. **Email Events Log** (`storage/logs/email-events.log`)
   - Email notifications (logged, not sent in dev)
   - Retention: 30 days

## Two-Factor Authentication (2FA)

### Implementation

**Library:** PragmaRX/Google2FA  
**QR Code:** BaconQrCode  
**Method:** Time-based One-Time Password (TOTP)

### Features

1. **Mandatory 2FA**
   - All admin users must enable 2FA before accessing dashboard
   - Enforced by `EnsureTwoFactorEnabled` middleware

2. **Setup Process**
   - Generate secret key (stored encrypted)
   - Display QR code for scanning
   - Verify with TOTP code
   - Generate 10 recovery codes
   - Confirm and save

3. **Recovery Codes**
   - 10 codes per user
   - Stored encrypted in database
   - Can be regenerated by user
   - Each code is 10 characters (uppercase hexadecimal)

4. **Challenge Flow**
   - User logs in with password
   - Redirected to `/two-factor-challenge`
   - Enter TOTP code or recovery code
   - On success, access granted
   - On failure, return to challenge with error

### Storage

```php
// User model fields
'two_factor_secret' => 'encrypted:string'
'two_factor_recovery_codes' => 'encrypted:json'
'two_factor_confirmed_at' => 'datetime'
```

### Security Considerations

✅ Secret keys encrypted at rest  
✅ Recovery codes encrypted at rest  
✅ Audit logging on enable/disable  
✅ QR code generated per session  
❌ Recovery codes not marked as used (see Missing Best Practices)  
❌ No rate limiting on 2FA challenge (see Missing Best Practices)

## Missing Best Practices

### Critical (Security Impact)

1. **❌ No Database Migrations for Security Tables**
   - **Issue:** `login_attempts` and `admin_activity_logs` tables are not defined in migration files
   - **Impact:** Application will fail to track attempts or log events without manual table creation
   - **Risk Level:** HIGH
   - **Recommendation:** Create migration files with proper schema

2. **❌ No Rate Limiting on 2FA Challenge**
   - **Issue:** Unlimited attempts on 2FA code entry
   - **Impact:** Brute-force attacks on 2FA codes (1,000,000 combinations)
   - **Risk Level:** HIGH
   - **Recommendation:** Implement rate limiting (e.g., 5 attempts per 15 minutes)

3. **❌ Recovery Codes Not Marked as Used**
   - **Issue:** Recovery codes can be reused multiple times
   - **Impact:** If a code is compromised, it remains valid indefinitely
   - **Risk Level:** MEDIUM-HIGH
   - **Recommendation:** Mark codes as used after first use, store used timestamps

4. **❌ No Password Strength Requirements**
   - **Issue:** Weak passwords accepted
   - **Impact:** Vulnerable to dictionary and brute-force attacks
   - **Risk Level:** MEDIUM-HIGH
   - **Recommendation:** Implement password rules (min 12 chars, uppercase, lowercase, numbers, symbols)

5. **❌ No CAPTCHA on Login Form**
   - **Issue:** Automated brute-force scripts can attempt logins
   - **Impact:** Easier to perform distributed attacks
   - **Risk Level:** MEDIUM
   - **Recommendation:** Add reCAPTCHA v3 or hCaptcha after 2-3 failed attempts

6. **❌ No Compromised Password Checking**
   - **Issue:** Users can set passwords from known breaches
   - **Impact:** Account takeover via credential stuffing
   - **Risk Level:** MEDIUM
   - **Recommendation:** Integrate HaveIBeenPwned API or similar service

### Important (Usability & Security)

7. **❌ No Email Verification**
   - **Issue:** `email_verified_at` column exists but not enforced
   - **Impact:** Users can register with fake/mistyped emails
   - **Risk Level:** LOW-MEDIUM
   - **Recommendation:** Implement email verification flow

8. **❌ No Session Timeout Configuration**
   - **Issue:** Sessions may persist indefinitely
   - **Impact:** Unattended sessions remain active
   - **Risk Level:** MEDIUM
   - **Recommendation:** Set session lifetime (e.g., 120 minutes) and idle timeout

9. **❌ No Notification for New Login Locations**
   - **Issue:** Users not alerted to logins from new IPs or locations
   - **Impact:** Account compromise detection delayed
   - **Risk Level:** MEDIUM
   - **Recommendation:** Email notification for logins from new IP addresses

10. **❌ No Password Expiration Policy**
    - **Issue:** Passwords never expire
    - **Impact:** Long-term compromise risk
    - **Risk Level:** LOW-MEDIUM
    - **Recommendation:** Optional password expiration (e.g., 90 days) for high-security environments

11. **❌ No Concurrent Session Management**
    - **Issue:** Same user can have unlimited active sessions
    - **Impact:** Shared credentials, account compromise harder to detect
    - **Risk Level:** MEDIUM
    - **Recommendation:** Limit concurrent sessions or show active sessions list

12. **❌ No Backup Authentication Method**
    - **Issue:** If user loses 2FA device and recovery codes, account lockout
    - **Impact:** Account recovery requires admin intervention
    - **Risk Level:** LOW-MEDIUM
    - **Recommendation:** Add backup email verification or security questions

### Advanced (Enhancement)

13. **❌ No Device Fingerprinting**
    - **Issue:** Only IP address is tracked
    - **Impact:** Device-level tracking unavailable
    - **Recommendation:** Implement device fingerprinting (browser, OS, screen resolution)

14. **❌ No IP Whitelist/Blacklist**
    - **Issue:** Cannot block known malicious IPs
    - **Impact:** Repeated attacks from same sources
    - **Recommendation:** Admin interface to manage IP allowlist/denylist

15. **❌ No Anomaly Detection**
    - **Issue:** No behavioral analysis
    - **Impact:** Sophisticated attacks undetected
    - **Recommendation:** ML-based anomaly detection (login times, locations, patterns)

16. **❌ No Progressive Delay on Failed Attempts**
    - **Issue:** Hard lockout after 5 attempts
    - **Impact:** Aggressive lockout may impact legitimate users
    - **Recommendation:** Exponential backoff (1s, 2s, 4s, 8s, lockout)

17. **❌ No Security Headers for Specific Vulnerabilities**
    - **Issue:** CSP allows `unsafe-inline` and `unsafe-eval`
    - **Impact:** XSS vulnerability surface increased
    - **Recommendation:** Tighten CSP, add nonce-based inline scripts

18. **❌ No Automated Security Testing**
    - **Issue:** Only basic tests exist
    - **Impact:** Regressions may go unnoticed
    - **Recommendation:** Add comprehensive security test suite

19. **❌ No Failed Login Attempt Cleanup**
    - **Issue:** `pruneOldAttempts()` method exists but not scheduled
    - **Impact:** Database bloat over time
    - **Recommendation:** Add to Laravel scheduler (daily cleanup of 30+ day old attempts)

20. **❌ No Password Reset Flow**
    - **Issue:** Password reset mentioned but not implemented
    - **Impact:** Users with 3+ lockouts cannot recover
    - **Recommendation:** Implement secure password reset with email verification

   // Migration for login_attempts table
   Schema::create('login_attempts', function (Blueprint $table) {
       $table->id();
       $table->string('email')->index();

       $table->uuid('id')->primary();

       $table->string('event_type')->index();
