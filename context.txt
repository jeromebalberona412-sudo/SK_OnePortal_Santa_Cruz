SK_ONEPORTAL: Main Context (Current + Future)

LAST UPDATED
- May 3, 2026

PROJECT OVERVIEW
- Product: SK_ONEPORTAL (web app ecosystem for Santa Cruz SK governance)
- Architecture: Modular monolith split into role-based Laravel apps
- Collaboration: Trunk-based development; each app keeps its own Laravel/composer setup
- Data model direction: Multi-tenant (tenant-scoped users and role data)

TARGET USER SCALE
- Admin: 4 users
- SK-Federations: 12 users
- SK-Officials: 12 officials x 26 barangays = 312 users
- Kabataan: age 13-30, 10k+ expected users across barangays

TECH STACK
- Backend: Laravel
- Frontend: Blade, HTML, CSS, JavaScript
- Database: Supabase/PostgreSQL (centralized, shared across all apps)
- Cache: Redis
- File storage: Cloudinary (for images/logos), Supabase Storage (planned for docs)
- Reporting: barryvdh/laravel-dompdf + phpoffice/phpword (SK_Officials)

SYSTEM VISION
SK_ONEPORTAL combines multiple systems in one platform:
1) Social network / community feed
2) Document/reporting system
3) Identity registry (Kabataan profiling)
4) Governance workflow system
5) Notification + engagement layer
6) Chatbot assistant (planned)

CURRENT IMPLEMENTED FEATURES (AS OF MAY 3, 2026)

Admin (Production-grade)
- Authentication: custom login/logout, Fortify, 2FA setup/challenge/recovery
- Access control: auth, ensure2fa, role-based middleware
- Account governance (SK-Fed and SK-Officials):
  - List, create, update, deactivate, reset password, extend term
  - Assign role, official profile, term, barangay
- Barangay Logos module: upload/manage per-barangay logos via Cloudinary
- Tenant bootstrap: resolves tenant context, seeds barangays when missing
- Audit/security: login/logout and governance audit events, lockout support
- Deleted accounts views: DeletedSkOfficials, DeletedSkFederation
- Archived Records module
- Contact Us module
- Password reset: routes and views exist; controller logic is placeholder-level

SK-Federations (Most complete security implementation)
- Authentication: login UI, Fortify authenticateUsing with custom auth service
- Security controls:
  - Role + tenant scope enforcement
  - Email verification flow with resend and signed links
  - Re-verification on changed/new device fingerprint
  - Single-session enforcement with heartbeat checks
  - Session takeover flow with OTP send/verify
  - Login attempt tracking, lockout, trusted-device registration,
    suspicious-login detection, auth audit logging
- Password reset: full end-to-end (forgot form, reset form, broker, token validation,
  tenant/role checks, complexity validation, reuse blocking, session invalidation,
  Turnstile middleware, rate limiting, custom reset email)
- Protected modules: Dashboard, Profile (under full auth + session middleware stack)
- Profile editing: currently returns disabled messages
- Community Feed module: reads announcements + reactions + comments
- Barangay Monitoring module
- Kabataan Monitoring module
- Reports module
- Archive module
- Barangay ABYIP module
- UX: shared loading screen assets, network error page

SK-Officials (Production-grade as of May 2026 — no longer demo-level)
- Authentication: full production security stack matching SK-Federations
  (trusted devices, login attempts, auth audit logs, feature flags,
   email verified devices, OTP, single-session, lockout)
- Modules with views/routes/controllers:
  Dashboard, Profile, Calendar, Committees, Programs, ABYIP, BudgetFinance,
  Kabataan (list/view), KKProfilingRequests (review workflow),
  PreviousKabataan (archive view), Announcement (create/list/view),
  ScheduleKKProfiling (create/manage schedules),
  schedule_programs, Rejected_KKProfiling, Rejected_Scholarship,
  Deleted_Kabataan, layout
- KK Profiling: SK Officials can schedule profiling windows (date_start, date_expiry,
  link, status enum) and review kabataan_registrations with evaluation_status/notes
- Announcements: SK Officials post announcements (type enum, title, body, image, link)
  visible in community feed; reactions and comments are polymorphic
- PDF/DOCX export: barryvdh/laravel-dompdf + phpoffice/phpword installed

Kabataan (Active development as of May 2026)
- Active modules: Authentication, Dashboard, Homepage, Profile, KKProfiling, Programs
- KK Profiling registration flow:
  - Multi-step state machine: pending_verification → email_verified → password_set → active
  - Form data stored as JSON (form_data column)
  - evaluation_status + evaluation_notes added for SK Officials review
  - RegistrationEvaluationService handles evaluation logic
  - KabataanVerifyEmail notification for email verification
- Dashboard: shows community feed (announcements from SK Officials)
- Homepage, Profile, Programs modules in place

CROSS-CUTTING COMPLEXITY AND RISKS
- Main project risk is scope/complexity, not core technology
- Hidden complexity areas:
  - Notifications across channels: email, realtime, in-app
  - Multi-role authorization and tenant boundaries
  - Large-scale file/document management
  - Reporting automation for LGU workflows

FILE STORAGE STRATEGY (DECISION)
- Do not store uploaded files on Laravel server disk
- Store IDs, images, reports, PDFs, DOCX in external object storage:
  - Current: Cloudinary (logos, images)
  - Preferred: Supabase Storage (documents, reports)
  - Alternative: S3

REPORTING ENGINE (HIGH PRIORITY)
- Build this early; strong differentiator for LGU compliance
- Required capabilities:
  - DOCX template engine (phpoffice/phpword installed in SK_Officials)
  - Auto-generated accomplishment reports
  - Yearly analytics and printable compliance outputs

EARLY FOUNDATION RECOMMENDATIONS
- Create a notification module early
- Standardize report templates and export pipeline early
- Define role/tenant permission matrix before social + posting rollout
