SK_ONEPORTAL: Main Context (Current + Future)

LAST UPDATED
- March 8, 2026

PROJECT OVERVIEW
- Product: SK_ONEPORTAL (web app ecosystem for Santa Cruz SK governance)
- Architecture: Modular monolith split into role-based Laravel apps
- Collaboration: Trunk-based development; each app keeps its own Laravel/composer setup
- Data model direction: Multi-tenant (tenant-scoped users and role data)

TARGET USER SCALE
- Admin: 4 users
- SK-Federations: 12 users
- SK-Officials: 12 officials x 26 barangays = 312 users
- Kabataan: age 13-30, 10k+ expected users across barangays

TECH STACK
- Backend: Laravel
- Frontend: Blade, HTML, CSS, JavaScript
- Database: Supabase (centralized)
- Cache: Redis

SYSTEM VISION
SK_ONEPORTAL combines multiple systems in one platform:
1) Social network
2) Document/reporting system
3) Identity registry
4) Governance workflow system
5) Notification + engagement layer
6) Chatbot assistant

CURRENT IMPLEMENTED FEATURES (AS OF MARCH 8, 2026)

Admin (Implemented, with some placeholders)
- Authentication:
	- Custom login/logout UI and flow
	- Fortify integration via custom authentication service
	- Two-factor setup/challenge/recovery flows
- Access control:
	- `auth`, `ensure2fa`, and role-based middleware for protected modules
- Core pages:
	- Authenticated dashboard and profile pages
- Account governance module (SK-Federations and SK-Officials accounts):
	- List by role with search, barangay filter, status filter, and pagination
	- Create account + assign role + create official profile + assign term/barangay
	- Update account/profile/term details
	- Deactivate account
	- Reset password (admin-triggered)
	- Extend official term
- Tenant bootstrap support:
	- Resolves tenant context and seeds barangays when missing
- Audit/security:
	- Login/logout and governance audit events
	- Login attempt and lockout support in auth services
- Password reset status in Admin app:
	- Forgot/reset routes and views exist
	- Controller logic is still placeholder/UI-level and not fully production-grade reset workflow

SK-Federations (Most complete security implementation so far)
- Authentication and login:
	- Login entry UI
	- Fortify authenticateUsing bound to custom authentication service
- Security controls:
	- Role + tenant scope enforcement
	- Email verification wait/notice/success flow with resend and signed links
	- Re-verification on changed/new device fingerprint
	- Single-session enforcement with heartbeat checks
	- Session takeover flow with OTP send/verify
	- Login attempt tracking, lockout handling, trusted-device registration,
		suspicious-login detection, and auth audit logging
- Password reset (implemented end-to-end):
	- Forgot password form and reset form/success views
	- Password broker link sending and token validation
	- Tenant/role checks for reset eligibility
	- Password complexity and confirmation validation
	- Password reuse blocking
	- Session invalidation after successful reset
	- Turnstile middleware and configurable rate limiting
	- Custom SK-Federations reset notification email
- Protected modules:
	- Dashboard route protected by `auth`, `verified`, `single.session`,
		`sk_fed.access`, `trusted.device`, and `prevent.back`
	- Profile page available under same protection
- Profile editing status:
	- Profile update and password update endpoints currently return disabled messages
- UX/reliability additions:
	- Shared loading screen assets (`/shared/css/loading.css`, `/shared/js/loading.js`)
	- Network error page and DB/network exception mapping in bootstrap exception handling

SK-Officials (Prototype stage; no longer scaffold-only)
- Has custom routes and screens for:
	- Login/forgot-password entry
	- Dashboard
	- Profile
	- Calendar
	- Announcements
- Current auth behavior is demo-level:
	- Uses simple session flag and hardcoded credentials in controller
	- Not yet connected to production-grade role/tenant/auth security stack

Kabataan (Current status)
- Base Laravel scaffold only
- Default welcome route present
- No production feature modules implemented yet

CROSS-CUTTING COMPLEXITY AND RISKS
- Main project risk is scope/complexity, not core technology
- Hidden complexity areas:
	- Notifications across channels: email, realtime, in-app
	- Multi-role authorization and tenant boundaries
	- Large-scale file/document management
	- Reporting automation for LGU workflows

FILE STORAGE STRATEGY (DECISION)
- Do not store uploaded files on Laravel server disk
- Store IDs, images, reports, PDFs, DOCX in external object storage:
	- Preferred: Supabase Storage
	- Alternative: Cloudinary or S3

REPORTING ENGINE (HIGH PRIORITY)
- Build this early; this can be a strong differentiator
- Required capabilities:
	- DOCX template engine
	- Auto-generated accomplishment reports
	- Yearly analytics and printable compliance outputs

EARLY FOUNDATION RECOMMENDATIONS
- Create a notification module early
- Standardize report templates and export pipeline early
- Define role/tenant permission matrix before social + posting rollout
